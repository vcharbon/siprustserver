SIP Rust Server — High-Availability replication

How call context survives a worker crash: the front-proxy load-balancer + OPTIONS health/readiness layer, and the peer-to-peer replication message exchange that keeps a backup current and re-hydrates a rebooted primary.

No Redis — each b2bua is both a replication server and a client. See ADR 0011 for the decisions and the plan file for the full wire spec + reconnect FSM.

Two replication directions, always name them: Forward = primary → backup (continuous; the backup pulls). Re-hydration = backup → rebooting primary (once, on boot). Reverse = acting-backup → original primary (after a takeover). Two generations: gen = incarnation (per worker-restart); call_gen = content version (last-writer-wins).

Part 1 — SIP-level HA: load-balancer proxy + OPTIONS

The stateless front proxy picks a primary + backup per call with rendezvous hashing, pins the choice in a signed Record-Route cookie, health-probes workers with OPTIONS, and routes around a dead/draining/not-ready worker to its backup.

1.1 — Cluster topology

Endpoints talk only to the proxy; the proxy load-balances across the b2bua StatefulSet; the b2buas replicate peer-to-peer. Membership comes from k8s.

1.2 — New call: HRW pick + stickiness cookie

The proxy hashes the Call-ID to a deterministic primary (and 2nd-best backup), stamps them into a signed cookie, and the primary replicates the call forward to the backup.

1.3 — Worker health state machine (proxy view)

The proxy classifies each worker from its OPTIONS reply. NotReady is "up but boot-replication drain unfinished"; Draining is SIGTERM; Dead is repeated probe timeout.

stateDiagram-v2
  [*] --> Unknown: registered, not yet probed
  Unknown --> Alive: OPTIONS 200
  Unknown --> Dead: probe timeout x threshold
  Alive --> NotReady: OPTIONS 503 not-ready
  NotReady --> Alive: OPTIONS 200 (re-hydrated + current)
  Alive --> Draining: OPTIONS 503 Retry-After 0 (SIGTERM)
  Draining --> Dead: grace expired
  Alive --> Dead: probe timeout x threshold
  Dead --> Unknown: pod respawn (fresh-pod guard)

1.4 — In-dialog routing decision

On each in-dialog request the proxy verifies the cookie HMAC, then routes by primary health — to the primary, or (on dead/draining/fresh-pod) to the backup that holds the replica.

1.5 — Crash → failover → reboot → reclaim (the whole SIP story)

b0 crashes mid-call; the proxy routes in-dialog traffic to b1 (which holds the replica and takes over); b0 reboots, stays NotReady until re-hydrated, then reclaims.

Part 2 — HA replication message exchange

The replication engine (repl-net transport + b2bua/src/repl): a compacted per-peer changelog, a push-after-subscribe pull stream, and a bootstrap scan — all message-granular so every frame is recordable.

2.1 — Dual-channel mesh (each worker is primary AND backup)

With HRW, calls split across the cluster, so every pair has two Forward channels. Bootstrap (dotted) flows the other way only when a primary reboots.

graph LR
  subgraph A["b2bua-0"]
    A_pri["pri:b0  (owns)"]
    A_bak["bak:b1  (backs up b1)"]
    A_log["changelog -> b1"]
  end
  subgraph B["b2bua-1"]
    B_pri["pri:b1  (owns)"]
    B_bak["bak:b0  (backs up b0)"]
    B_log["changelog -> b0"]
  end
  A_log -->|"Forward: b0's calls"| B_bak
  B_log -->|"Forward: b1's calls"| A_bak
  B_bak -.->|"re-hydrate on b0 reboot"| A_pri
  A_bak -.->|"re-hydrate on b1 reboot"| B_pri

2.2 — Steady-state Forward replication (push-after-subscribe)

The backup subscribes once; the primary pushes Data as calls mutate and a Noop when the changelog drains. The body is read from the store (zero-copy Arc) at send time — never re-encoded.

sequenceDiagram
  autonumber
  participant B as b2bua-1 (backup / puller)
  participant A as b2bua-0 (primary / server)
  participant LOG as A changelog -> b1
  participant ST as A CallStore (Arc bytes)
  B->>A: PullRequest(Replog, since=(gen,ctr), chunk=128)
  loop entries with (gen,counter) > since
    A->>LOG: next ref at cursor
    A->>ST: get_call(ref) -> Arc clone (brief lock)
    A-->>B: Data(gen, ctr, op, partition=bak, call_ref, call_gen, body)
    B->>B: apply iff newer than W; LWW by call_gen; advance W
  end
  A-->>B: Noop(head)
  Note over B: current = true (sticky) -> feeds readiness gate
  Note over A,B: connection stays open; A keeps PUSHING new Data live
  B->>A: Ack(up_to) periodic -> A trims retention

2.3 — Write path: non-blocking append, compaction, drain

The call handler bumps the changelog non-blocking (never waits on a subscriber). The drain task reads the body under a brief lock then writes on owned bytes — no lock held across I/O, survives the call being removed mid-send.

graph TD
  H["call handler (per-call FIFO)"] -->|mutate Call| F["flush: encode once -> Arc bytes"]
  F -->|put_call| RCS["ReplicatingCallStore"]
  RCS -->|store Arc body| ST[("CallStore  pri:self")]
  RCS -->|"try_send: move ref to new counter (non-blocking)"| CL[("per-peer changelog\ncompacted by callRef")]
  CL -. notify .-> D["drain task (per subscriber)"]
  D -->|"get_call -> Arc clone (brief lock)"| ST
  D -->|"write owned Arc (no lock held)"| SOCK(["TCP socket / sim"])
  CL --- N["compaction: re-update X moves entry\ncounter 5 -> 22, old dropped\nmemory ~ live call count"]

2.4 — Re-hydration / bootstrap (rebooting primary pulls itself back)

b0 reboots empty and scans b1's bak:b0 partition. b1 snapshots the keys (brief lock) and captures the head watermark at scan start, then streams bodies lazily. b0 seeds that watermark and tails — so any takeover mutation b1 made during the scan is re-delivered.

sequenceDiagram
  autonumber
  participant A as b2bua-0 (rebooting primary)
  participant B as b2bua-1 (backup / server)
  Note over A: boot: memory wiped, NotReady
  A->>B: PullRequest(Bootstrap, caller=b0)
  Note over B: snapshot bak:b0 KEYS (brief lock)
  Note over B: W = changelog head AT SCAN START
  loop ~128 keys per batch
    B->>B: read current body (short per-call lock)
    B-->>A: Data(op=create, partition=pri, call_ref=b0|..., body)
    A->>A: import as pri:b0
  end
  B-->>A: Noop(W)  [bootstrap terminal]
  Note over A: seed Replog watermark = W
  A->>B: PullRequest(Replog, since=W)  [same connection]
  B-->>A: Data... takeover deltas (counter > W) ... Noop
  Note over A: re-hydrated + current -> answer OPTIONS 200 Ready

2.5 — Puller reconnect state machine (per peer)

Covers TCP death, exponential backoff, topology-driven park/resume (k8s pod gone/back), and the watermark-retained reconnect. A pod reboot bumps gen, so frames apply without a manual reset.

stateDiagram-v2
  [*] --> Connecting: topology adds peer
  Connecting --> Bootstrapping: ok + self cold-start/reset
  Connecting --> Tailing: ok + self has state (TCP blip)
  Connecting --> Backoff: connect fails
  Bootstrapping --> Tailing: terminal Noop, seed W
  Tailing --> Tailing: Data apply / Noop sets current
  Tailing --> Bootstrapping: ResetToBootstrap
  Tailing --> Backoff: recv None / send fail (peer crash)
  Bootstrapping --> Backoff: recv None / send fail
  Backoff --> Connecting: backoff elapsed
  Tailing --> Parked: topology removes peer (retain W)
  Backoff --> Parked: topology removes peer
  Parked --> Connecting: topology re-adds (maybe new addr)

2.6 — Frame reference

Each message is a positional-msgpack array, tag-discriminated by element 0 (ADR-0008 ethos). Real TCP delimits with a 4-byte big-endian length prefix; the sim delivers whole messages.

MessageDirPositional fieldsPurpose
PullRequestclient → server [0, proto_ver, caller, mode, since_gen, since_counter, chunk]
mode: 0=Replog, 1=Bootstrap		 Open a subscription (tail) or a bootstrap scan. since_* ignored for Bootstrap.
Ackclient → server [1, caller, up_to_gen, up_to_counter] Optional; lets the server trim changelog retention.
Dataserver → client [2, gen, counter, op, partition, call_ref, call_gen, body_ttl_ms, indexes, body]
op: 0=create 1=update 2=delete; partition: 0=pri 1=bak		 One replicated call mutation. body = positional-msgpack Call (nil for delete/expired). Apply iff (gen,counter) beats the watermark; LWW by call_gen.
Noopserver → client [3, gen, counter] Caught-up heartbeat / bootstrap terminal — sets the sticky current flag.
ResetToBootstrapserver → client [4, reason] The watermark fell off the compacted tail; the client must re-pull from scratch.
Reboot / incarnation: a rebooted worker serves under a higher gen with the counter reset to 0, so (new_gen, 0) > (old_gen, *) — pullers apply the fresh stream without an explicit reset. A delete missed during a disconnect self-evicts via the call ttl (body_ttl_ms).

Glossary

Forward replication
primary → backup; continuous; the backup pulls. Source: the compacted changelog.
Re-hydration (bootstrap)
backup → rebooting primary; once, on boot; primary pulls back the calls it owns.
Backup re-subscription
peer → rebooted backup; Forward replication starting cold from (gen, 0).
Reverse replication
acting-backup → original primary; after a takeover, so the primary picks up the latest on return.
Incarnation gen
per worker-restart; the high word of the (gen, counter) watermark.
callGen
per-call content version (CallTopology.gen); resolves conflicts last-writer-wins.
Watermark
(gen, counter) pull cursor; retained forever per ordinal across disappear/reappear.
current flag
sticky per-peer "caught up to head" signal, set on a Noop; gates readiness.
element
a Call / call-context replica — the unit a backup holds.
Readiness states
NotReady (boot drain unfinished) → Ready → Draining (SIGTERM).

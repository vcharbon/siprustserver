SIP Rust Server — High-Availability replication

How call context survives a worker crash or partition: a peer-to-peer pull protocol — no Redis — that keeps a backup current and re-hydrates a rebooted primary. Each b2bua is both a replication server (serves its changelog) and a client (pulls its peers).

This is the model after ADR 0014: two single-flow sockets per peer, a per-call (p,b) version vector separate from the changelog watermark, and reactive-only takeover with push-based self-release. See ADR 0014 for the decisions and CONTEXT.md for the glossary.

1 · The two flows — two sockets per peer

The old single connection multiplexed both directions behind one watermark (and a Bootstrap→Replog two-request dance). It conflated the global changelog position with the per-call (p,b) and caused a re-hydration watermark collision. It is gone.

From a node N's view, each peer pair runs two independent single-flow sockets to one listen port — distinguished only by the partition on the opening PullRequest. Same FSM, parameterised by {store partition, arm-timers?, role}; one shared reconnect path on a cut.

Reclaim vs. Backup — what each flow pulls, where it lands, what it arms.

Reclaim calls N is primary for, that a peer holds as backup Pri pri:{N} arm (smoothed) primary

Both flows are permanent and have the same shape — a store-scan bootstrap followed by an endless tail:

  • Reclaim = bootstrap(reclaim my own calls a backup holds) → tail(catch a post-partition reverse-flush a live-but-partitioned primary missed). Its frames materialise into the live serving map: the bootstrap as a smoothed bulk re-arm, a tail straggler immediately.
  • Backup = bootstrap(load the peer's calls I back up) → tail(forward updates). Pure storage; arms nothing.

2 · One changelog, two partition-filtered cursors

Watermark is a changelog position. (p,b) is a per-call version vector. They are kept rigorously separate — conflating them was the "two-generations trap."

A node keeps one changelog: a single node-global (gen, counter) space under an injected incarnation gen (bumped per restart). Internally it splits into per-partition compacted sub-logs — one counter → callRef BTreeMap plus a by_ref index and a retained_floor, one each for Pri and Bak. A bump does an O(log L) move (drop the ref's old counter, insert at the new max), so each sub-log stays bounded to its live set. Each flow drains only its sub-log from its own cursor — no cross-partition filtering.

Watermark W = (gen, counter)
Purely a changelog position, ordered lexicographically with gen high. Per (peer, flow), retained by the supervisor across Park/disconnect/re-add. Advances only on a Noop(head) and on a post-bootstrap tail Data.at — never on a bootstrap frame (those all share at = W = head@scan-start). It is never read from or written to any call's (p,b).
Version vector (p, b)
Per call. p is bumped only by the call's primary on a local mutation; b only by an acting-backup on a takeover mutation. The sole idempotency gate on apply (see §5). There is no client-side at ≤ W apply-gate — dropping it is what fixed the bootstrap-frame collision.
Bootstrap = store scan
Each flow's bootstrap is a store scan, not a changelog replay: a created-then-quiescent call's changelog entry compacts away, so only the scan is a complete snapshot. It captures W = head at scan start, streams every live body stamped at that W, then tails from W.
Loopback invariant
Applying a received Data frame mutates the store but never appends to the local changelog (the inbound apply uses peer: None). Only a locally-originated mutation appends — so a change N received as a backup can never be re-served back to the peer that sent it. The puller's private cursor still advances; the changelog does not.

3 · The unified server send loop

One mechanism for both flows: a pure sleep-poll. The old Notify/subscriber/Subscription machinery is deleted.

A node listens once and serves each accepted connection one flow (the opening PullRequest.partition selects it). The serve routine is parameterised by (partition, caller):

  1. Reset check. If a warm since fell below this sub-log's compacted retained_floor, send ResetToBootstrap and stop — the puller re-pulls from (0,0).
  2. Bootstrap (cold only, since == (0,0)). Scan the flow's keyspace under brief locks — Reclaim scans bak:{caller}, Backup scans pri:{self} filtered to the calls caller backs up — capture W = head, stream each live body as Data{op: Put, at: W} in chunks (one write+flush per chunk, never a lock held across the socket). The tail then resumes warm from W, so scan and tail never double-deliver.
  3. Poll-tail. Every ~100 ms drain the sub-log range (since,..] taking limit+1 (= 501). While > 500 pending, stream Data back-to-back with no Noop and no sleep (each Data carries its own at, so the cursor rides the data and the TCP buffer fills). At ≤ 500, flush the remainder and sleep.

A Noop is emitted only on the catch-up edge (just drained a real backlog to head — the first such Noop sets the puller's sticky current flag) and on a ~20 s idle floor (liveness keepalive). A steady trickle needs no Noop. Bodies are read live at send time, so a slow puller always gets latest-per-call. The serve side is independent of local readiness: a NotReady node still answers its peers' pulls.

4 · Boot order & readiness — Reclaim first

A booting node re-hydrates its own partition before it takes on backup duty for others.

  • The supervisor opens the Reclaim streams immediately.
  • Ready ⇔ every Reclaim stream to a reachable peer hit its first Noop — bounded by a per-flow hard timer so an unreachable or pathologically-slow peer cannot hang readiness (liveness over completeness). A cold node with no calls reaches Ready at once (empty scan → immediate Noop).
  • The Backup streams open only after Ready. They are fire-and-forget and never gate readiness — observable via the store and a received-frame counter, not a readiness sub-state. (The gate is bandwidth prioritisation only; (p,b) makes any incidental overlap safe.)

All readiness introspection — all_current, is_current, bootstrap_complete, the retained watermark — scopes to the Reclaim flow. The Backup flow's current is metrics-only.

5 · Apply: (p,b) merge, takeover & self-release

Every Data is applied under the version-vector rule — the only idempotency. A stale or re-delivered frame is rejected by dominance, not by a watermark compare.

Let the incoming frame carry (p, b) and the local copy hold (sp, sb):

Delete
Apply unconditionally — delete-wins, both directions.
Forward / Bootstrap Put (the Backup flow, and any flow's pre-Noop scan)
Apply unless dominated: skip iff sp ≥ p && sb ≥ b (an idempotent re-delivery). The follower defers to the authority but never regresses. This rule is what makes every bootstrap frame — all sharing at = W — safe to apply without a watermark gate.
Reverse Put (the Reclaim flow's steady tail)
A backup reverse-flushed one of our calls. Apply iff we have not mutated past the backup's branch point and it genuinely advanced: p == sp && b > sb (or no local copy yet → accept; the reactive reclaim materialises it). Else keep our own.

Reactive-only takeover

There is no eager, death-triggered takeover. A quiescent failed-over dialog is recovered only when the rebooting primary reclaims it (the smoothed Reclaim bootstrap), or earlier when in-dialog traffic the load-balancer reroutes to a survivor triggers a reactive takeover: the survivor hydrates the call from its bak: replica, serves the transaction, and bumps b. A quiescent call on a permanently-lost node dies after the keepalive slack — the deliberate price of killing the eager-takeover stale-CSeq storm.

Push-based self-release

An acting-backup sheds its live takeover copy when the transaction(s) it served reach a terminal state — driven by the txn layer, never polled. The router, on a fresh takeover, arms a watch; when the last transaction for the watched call leaves the txn map, the layer emits CallQuiesced{call_ref} and the router self-releases (drop the live copy + cancel timers/txns + poison the dispatch queue), re-checking under the per-call lock. It propagates no delete: the bak: replica and the reverse-flushed (p,b) deltas remain, so the call lives on at its reclaiming primary. ("Last transaction left the map" — not "final response" — means a 2xx INVITE naturally releases at Timer H, after the ACK is relayed.) This replaces the removed Deactivate watermark handback.

No time-based settle anywhere. Reconciliation is (p,b)-causal: a partition may route a dialog to the backup for any duration, so correctness must not depend on a timer or settle window. Keepalive catch-up smoothing on reboot is performance-only and lives in the reclaim handler, never the timer driver.

6 · The wire message set — four frames

Positional msgpack: each frame is an array with an integer tag at element 0; field order is the contract. Pre-production, so tags renumber and the proto version bumps freely (currently v3).

FrameDirFieldsRole
PullRequestC→S proto_ver, caller, partition, since Open one flow. Bootstrap is implicit when since == (0,0) (scan-then-tail); else tail from since. One socket = one flow; there is no second request.
DataS→C at, op, partition, call_ref, p, b, body_ttl_ms, indexes, body? One mutation. op ∈ {Put, Delete} — Create and Update are merged into one idempotent Put (carries a body); Delete carries none.
NoopS→C at Catch-up edge (the first one ⇒ current) and the ~20 s idle keepalive. Advances the puller's watermark.
ResetToBootstrapS→C reason since fell below the compacted tail → discard W, re-pull from (0,0).

Removed (no vestige remains): Ack (a no-op retention hint — retention is now bounded by time/size, and a too-slow puller re-bootstraps via ResetToBootstrap); the PullMode enum and the Bootstrap/Replog two-request handshake (collapsed into the since == (0,0) rule); the Deactivate handback and eager takeover.
